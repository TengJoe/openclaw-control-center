# Project Structure

This document is the quickest way to orient yourself inside the repository before touching `server.ts` or adding new runtime features.

## Top Level

| Path | Purpose | Notes |
| --- | --- | --- |
| `src/` | Product source code | Main place for feature work and refactors |
| `scripts/` | Operational scripts and smoke checks | Prefer cross-platform Node/TS scripts for shared workflows |
| `test/` | Automated verification | Route, adapter, render, and snapshot coverage |
| `docs/` | Operator and maintainer docs | Setup, runbooks, architecture, structure notes |
| `runtime/` | Local generated artifacts | Mutable data, exports, logs, snapshots; not source-controlled product logic |
| `dist/` | TypeScript build output | Generated by `npm run build` |

## Source Layout

### `src/ui/`
- HTTP server, HTML rendering, route wiring, page-level caches
- `src/ui/server.ts` is still the shell entry for the UI, but it should stay focused on request orchestration and rendering glue
- When logic becomes reusable or stateful, move it down into `src/runtime/` or `src/clients/`

### `src/runtime/`
- Read models, derived view models, file-backed stores, search helpers, replay indexing, notification logic
- Best home for anything that answers: "given current local state, what should the UI show?"
- If a function can be tested without booting the whole server, it probably belongs here

### `src/clients/`
- Thin wrappers around OpenClaw interfaces and local fallbacks
- Best home for CLI / file / gateway integration boundaries
- Keep policy-free and transport-focused when possible

### `src/adapters/`
- Cross-layer adapters that assemble multiple client/runtime calls into a higher-level read model
- Good place for readonly snapshot composition and compatibility shims

### `src/mappers/`
- Translation between raw upstream payloads and typed internal summaries
- Keep formatting and coercion here instead of spreading ad-hoc parsing across the UI

### `src/contracts/`
- Internal request/response shapes for tool clients and local APIs
- Treat as stability boundaries when changing adapters or route payloads

## Scripts Layout

### `scripts/`
- Prefer Node/TS scripts over shell when the workflow should run on macOS, Linux, and Windows
- Good fits:
  - smoke tests
  - validation helpers
  - export/import tooling
  - watchdog or health utilities
- Keep shell only when the task is intentionally Unix-only and tiny

## Tests Layout

### `test/`
- `ui-render-*` and route tests: HTML shell and API wiring
- `usage-cost`, `office-*`, `session-*`: runtime logic and view-model coverage
- command-path tests: backup, prune, watchdog, heartbeat, and other operational flows

When adding a feature:
1. Add pure logic tests first if the behavior lives in `src/runtime/` or `src/clients/`
2. Add route/render coverage only for user-visible wiring
3. Avoid testing giant HTML blobs when a smaller derived function can be asserted instead

## Runtime Boundaries

### `runtime/`
- Generated, mutable, environment-specific
- Examples:
  - `runtime/last-snapshot.json`
  - `runtime/projects.json`
  - `runtime/tasks.json`
  - `runtime/exports/`
  - `runtime/*.log`

Do not move product logic here. Runtime files are data sources, not the implementation layer.

## Placement Rules

Use these rules when deciding where new code should live:

1. If it talks to OpenClaw CLI, local session files, or an upstream surface, put it in `src/clients/`
2. If it derives dashboard state from snapshots or stores, put it in `src/runtime/`
3. If it stitches multiple layers into one readonly surface, put it in `src/adapters/`
4. If it only formats or normalizes payloads, put it in `src/mappers/`
5. If it only handles HTTP or HTML composition, keep it in `src/ui/`
